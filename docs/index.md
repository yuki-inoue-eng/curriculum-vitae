## 警備ロボットのクラウド管理サービスの開発（準委任契約） [2023/08 〜 2025/06]

### 主な成果

#### エレベータ連動機構の挙動解析と仕様整理
設計者の退職によりブラックボックス化していたロボットのエレベータ乗降プロセスを解析しました。  
ロボット・エレベータ GW・イベント処理マイクロサービス間で発生するイベントフローを調査し、Job の実行タイミングや状態遷移を整理しました。  
解析結果を基にシーケンス図を作成し、チーム内で共有しました。  
また、イベント駆動で実行される Job の処理内容および発火タイミングを見直し、乗降プロセスの効率化を実施しました。  
その結果、エレベータドアの開扉ロック時間を約 30% 短縮し、大口顧客の離脱防止に貢献しました。

#### フロア移動経路探索機能のマイクロサービス移行
ロボット指示管理 API 内で実装されていたフロア移動の経路探索機能を、専用の経路探索マイクロサービスへ移行するプロジェクトに参画しました。  
経路探索マイクロサービス自体は別チームが開発し、私は既存システム側の移行設計および関連マイクロサービスとの連携改修を主導しました。  
フロア移動プロセスには複数のマイクロサービスが関与していたため、既存 API との互換性を維持しながらインターフェースを整理し、新しい経路探索 API に対応する形で関連サービスの呼び出し処理を修正しました。  
移行に伴う影響を最小化するため、Blue-Green デプロイを前提とした移行戦略を設計しました。各サービス間のインターフェース依存関係を整理し、互換性を保ったまま段階的に切り替え可能なデプロイ順序を設計しました。  
ロボット指示管理 API 内では、探索結果のキャッシュによる経路探索 API 呼び出し回数の削減や、目的地・現在地の保持などのフロア移動関連処理を Service レイヤーに集約することで、他の機能からも再利用可能な形に抽象化しました。  
最終的にはシミュレーション環境および実機検証を通して正常系・異常系の確認を行い、大きな障害なく新経路探索システムへの移行を実施しました。

## AbemaTV 検索基盤リプレイス（準委任契約） [2022/08 〜 2023/07]

### プロジェクト概要
AbemaTV におけるコンテンツ検索基盤のリプレイスおよびオンプレミスから GCP への移行プロジェクトに参画しました。  
Solr から Elasticsearch への移行、および検索 API・索引作成バッチの Java から Go への再実装を含む技術刷新を担当しました。  
初期メンバーとして参画し、先方 PM と 2 名体制で、設計・実装・検証・リリースまで一貫して推進しました。

| API        | gRPC, REST                                     |

### 主な成果

#### 検索 API と索引作成バッチの再設計・実装
旧検索基盤では、モデルの風化やコンテキスト境界の未定義、レイヤ分離が不十分な密結合設計により、変更容易性およびテスト容易性が低い状態となっていました。これらの課題を解消するため、Clean Architecture をベースとした設計へ再構築しました。  
概念モデル図を用いてドメイン整理を行い、ドメイン層とインフラ層の依存関係を明確化することで、保守性・拡張性およびテスト容易性の高いアプリケーション構造を実現しました。  
上記設計に基づき、既存の Java 実装であった検索 API および索引作成バッチを Go で再実装しました。  
再実装にあたってアプリケーションのレイヤ構成および実装パターンを定義し、初期実装を主導することで、後続開発者が同一の設計方針で開発を進められる開発基盤を構築しました。  
また、索引作成バッチでは、コンテンツデータおよびメタデータ取得処理の並列化を行い、処理時間を 85% 削減（20 分 → 3 分）しました。

#### Elasticsearch 運用基盤設計
検索基盤として採用した Elasticsearch の運用基盤の選定と運用方針の策定を行いました。  
まず運用基盤として ECK（Elastic Cloud on Kubernetes）を候補に技術検証を実施し、PoC 環境を構築して本番運用を想定した検証を行いました。  
その結果、Kubernetes 上での Elasticsearch 運用には高い運用コストと専門知識が必要であることが判明したため、チームの運用体制を踏まえ、Elastic Cloud（マネージドサービス）を採用する方針としました。  
また、破壊的変更を含む Elasticsearch のバージョン更新に備えて、Blue-Green デプロイを用いた安全なアップデート戦略を設計し、バージョン更新時の具体的な作業手順を策定・ドキュメント化しました。  
さらに、既存の Solr スキーマをベースに、再設計されたドメインモデルに対応する形で Elasticsearch の index / mapping を設計しました。  
index 設定の管理には Elastic 社提供の Terraform Provider を利用し、宣言的な設定管理を実現しました。

#### 負荷試験・パフォーマンスチューニング
過去のアクセス実績に基づき、最大 2,000 rps 程度の検索リクエストが発生することを想定して、k6 を用いた負荷試験を実施しました。  
CDN によるレスポンスキャッシュおよび Elasticsearch のクエリキャッシュなどに依存しない、純粋な検索処理のスループットを測定できるよう負荷試験シナリオを設計しました。  
また、Prometheus / Grafana により API および Elasticsearch のメトリクスを可視化し、負荷時のレイテンシ、スループット、CPU / メモリ使用率を継続的に観測できる状態を整備しました。  
これによりボトルネックを定量的に分析し、shard 数・replica 数およびノードスペックの構成決定に反映しました。

#### CI/CD パイプラインの構築
検索 API および索引作成バッチの Docker によるコンテナ化と Kubernetes マニフェストの作成を行いました。  
CircleCI を用いたテスト・ビルド・コンテナイメージの push を行う CI パイプラインを構築し、PipeCD を用いて本番環境および開発環境へのデプロイフローを整備しました。
