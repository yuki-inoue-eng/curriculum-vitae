## 警備ロボットのクラウド管理サービスの開発 (準委任契約) [2023/08 〜 2025/06]
### プロジェクト概要
警備ロボットのクラウド管理サービスの開発および機能改善

### プロジェクト詳細
警備ロボットのクラウド管理サービスにおいて、ロボットのフロア移動制御、エレベータ連動、認証基盤、管理機能の開発および既存機能の改善を担当

#### 使用技術

| 使用技術   | 内容                   |
|--------|----------------------|
| 言語     | Go, Ruby, TypeScript |
| クラウド   | AWS                  |
| データベース | PostgreSQL           |

| 監視     | Datadog              |

### 主な成果
#### エレベータ連動機構の挙動解析と仕様整理
設計者の退職によりブラックボックス化していたロボットのエレベータ乗降プロセスを解析しました。
ロボット・エレベータGW・イベント処理マイクロサービス間で発生するイベントフローを調査し、Job の実行タイミングや状態遷移を整理しました。
解析結果を基にシーケンス図を作成し、チーム内で共有しました。
また、イベント駆動で実行される Job の処理内容および発火タイミングを見直し、乗降プロセスの効率化を実施しました。
その結果、エレベータドアの開扉ロック時間を約30%短縮し、大口顧客の離脱防止に貢献しました。

#### フロア移動経路探索機能のマイクロサービス移行
ロボット指示管理 API 内で実装されていたフロア移動の経路探索機能を、専用の経路探索マイクロサービスへ移行するプロジェクトに参画しました。
経路探索マイクロサービス自体は別チームが開発しており、私は既存システムからの移行および関連マイクロサービスとの連携部分の改修を担当しました。
フロア移動プロセスには複数のマイクロサービスが関与していたため、既存 API との互換性を維持しながらインターフェースを整理し、新しい経路探索 API に対応する形で関連サービスの呼び出し処理を修正しました。
移行に伴う影響を最小化するため Blue-Green デプロイを前提とした移行戦略を設計しました。各サービス間のインターフェース依存関係を整理し、互換性を保ったまま段階的に切り替え可能なデプロイ順序を設計しました。
ロボット指示管理 API 内では探索結果のキャッシュによる経路探索 API 呼び出し回数の削減や、目的地・現在地の保持などのフロア移動関連処理を Service レイヤーに集約することで、他の機能からも再利用可能な形に抽象化しました。
最終的にはシミュレーション環境および実機検証を通して正常系・異常系の確認を行い、大きな障害を発生させることなく新経路探索システムへの移行を完遂しました。

#### GraphQL を用いた管理機能開発
GraphQL を用いた管理系機能の追加開発を担当。
主に CRUD ベースのバックエンド実装を担当し、一部機能ではフロントエンド実装まで対応しました。
- ロボットのパーツ交換履歴管理機能
- ロボットのシステムアナウンス設定管理機能
- ロボット画像コンテンツ管理機能
- その他管理系機能の追加・改修

### プロジェクト概要
AbemaTV におけるコンテンツ検索基盤のリプレイスおよびクラウド移行

移行に伴う主要技術の変更は以下のとおりです。
先方企業の PM と 2 名体制で、設計・実装・検証・リリースまで検索基盤リプレイスの主要工程を担当しました。

### 主な成果
#### 検索基盤のドメインモデルおよびアーキテクチャ再設計
旧検索基盤では

- モデルの風化
- コンテキスト境界の未定義
- テスト困難な密結合設計

といった問題が存在していました。これらの課題を解消するため、DDD と Clean Architecture をベースとした設計へ再構築しました。
概念モデル図を用いてドメイン整理を行い、保守性・拡張性の高いアプリケーション構造へ改善しました。

#### 検索 API と索引作成バッチの設計・実装
既存システムで Java で実装されていた検索 API および索引作成バッチを Go で再実装しました。
既存システムではアプリケーションのレイヤ分離が不十分でテストが困難な構造となっていたため、ドメイン層とインフラ層の依存関係を整理し、単体テストを実装可能な構成へ改善しました。
また初期実装の大部分を担当し、アプリケーションのレイヤ構成や実装パターンを整理することで 後続の開発者が同様の設計方針で機能実装を行える開発基盤を構築しました。
さらに索引作成バッチでは コンテンツデータおよびメタデータ取得処理の並列化を行い、処理時間を 20 分から 3 分へ短縮しました。

#### Elasticsearch 運用基盤設計
検索基盤として採用した Elasticsearch の運用基盤の選定と運用方針の策定を担当しました。
まず運用基盤として ECK（Elastic Cloud on Kubernetes） を候補に技術検証を実施し、PoC 環境を構築して本番運用を想定した検証を行いました。
その結果、Kubernetes 上での Elasticsearch 運用には高い運用コストと専門知識が必要であることが判明したため、チームの運用体制を踏まえ Elastic Cloud（マネージドサービス）の採用を提案しました。
また Elasticsearch のバージョン更新には破壊的変更が含まれるため、Blue-Green デプロイを用いた安全なアップデート戦略を設計し、バージョン更新時の具体的な運用手順を策定・ドキュメント化しました。
さらに既存の Solr スキーマをベースに、再設計されたドメインモデルに対応する形で Elasticsearch の index / mapping を設計しました。
index 設定の管理には Elastic 社提供の Terraform Provider を利用し、宣言的な設定管理を実現しました。

#### 負荷試験・パフォーマンスチューニング
過去のアクセス実績に基づき、最大 2000 rps 程度の検索リクエストが発生することを想定して k6 を用いた負荷試験を実施しました。
CDN によるレスポンスキャッシュおよび ES のクエリキャッシュなどに依存しない純粋な検索処理のスループットを測定できるよう負荷試験シナリオを設計しました。
検証結果を基に shard 数・replica 数およびノードスペックの構成を決定しました。

#### CI/CD パイプラインの構築
検索 API および索引作成バッチの Docker によるコンテナ化と Kubernetes マニフェストの作成を行いました。
CircleCI を用いたテスト・ビルド・コンテナイメージの push を行う CI パイプラインを構築し、PipeCD を用いて本番環境および開発環境へのデプロイフローを整備しました。
